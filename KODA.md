# KODA.md — Контекст проекта стенда машинного зрения

Этот файл содержит структурированную информацию о проекте для использования в качестве инструкционного контекста при взаимодействии с ИИ-агентом.

## 📌 Обзор проекта

**Название:** Стенд машинного зрения (MV Service)  
**Версия:** 4.5  
**Последнее обновление:** Апрель 2026

### Назначение

Автоматизированная система контроля качества деталей на производстве:

1. **Обнаружение детали** — датчики фиксируют появление объекта на конвейере
2. **Позиционирование** — деталь останавливается под камерой
3. **Съёмка и анализ** — камера делает снимок и определяет качество
4. **Сортировка** — деталь перемещается к выходу, результат индицируется лампами
5. **Сохранение результатов** — все проверки записываются в базу данных

**Результаты контроля:**
- ✅ **OK (годен)** — зелёная лампа, деталь движется к датчику Д4
- ❌ **NG (брак)** — красная лампа, деталь движется к датчику Д3

## 📡 Сценарии работы

### Сценарий A — Непрерывный контроль

- Камера выполняет периодические измерения (интервал настраивается)
- При обнаружении **NG** конвейер останавливается, загорается красная лампа
- **OK** — конвейер продолжает работу

### Сценарий B — Одиночный снимок

- По фронту тумблера B (DI5) выполняется одно измерение
- Результат индицируется лампами на 2 секунды

### Сценарий C — Циклический контроль (основной)

```
IDLE → (спад Д1) → CONVEYOR_RUNNING_TO_CAMERA → (спад Д2) → POSITIONED
→ (съёмка) → CONVEYOR_RUNNING_TO_EXIT → (спад Д3/Д4) → IDLE
```

**Логика:**

1. **IDLE:** Ожидание детали на входе (спад DI0)
2. **CONVEYOR_RUNNING_TO_CAMERA:** Конвейер работает до позиции под камерой (спад DI1)
3. **POSITIONED:** Деталь остановлена, выполняется измерение
   - **OK:** Лампа зелёная, движение до DI4
   - **NG:** Лампа красная, движение до DI3
4. **CONVEYOR_RUNNING_TO_EXIT:** Движение до целевого датчика
5. **IDLE:** Возврат в начало

## 🔌 Схема подключений ОВЕН МК210-312

### Входы (DI)

| Порт | Назначение | Тип |
|------|------------|-----|
| DI0 | Датчик Д1 — вход конвейера | NC |
| DI1 | Датчик Д2 — под камерой (левый) | NC |
| DI2 | Датчик Д3 — под камерой (правый) | NC |
| DI3 | Датчик Д4 — выход конвейера | NC |
| DI4 | Тумблер A — выбор сценария | — |
| DI5 | Тумблер B — выбор сценария | — |

### Выходы (DO)

| Порт | Назначение |
|------|------------|
| DO0 | Конвейер (пуск/стоп) |
| DO1 | Толкатель брака (резерв) |
| DO2 | Лампа NG (красная) |
| DO3 | Лампа OK (зелёная) |

### Выбор сценария (тумблеры)

| Тумблер A | Тумблер B | Сценарий |
|-----------|-----------|----------|
| 1 | 0 | A (непрерывный контроль) |
| 0 | 1 | B (одиночный снимок) |
| 0 | 0 | C (циклический контроль) |
| 1 | 1 | — (не используется) |

## 🛠 Правила разработки

### Стиль кодирования

- **Форматирование:** Стандартный Python (4 пробела, no tabs)
- **Импорты:** Группировка — стандартные библиотеки, затем сторонние, затем локальные
- **Документация:** Docstrings для классов и публичных методов
- **Имена:** snake_case для функций/переменных, PascalCase для классов
- **Логирование:** Использовать `utils.logger.setup_logger()`, не print()

### Поток данных

1. **Контроллер** запускается в отдельном потоке при старте приложения
2. **Веб-запросы** не блокируют основной цикл — отправляют команды через `Queue`
3. **Оборудование** подключается асинхронно, при потере связи — автоматическое переподключение с экспоненциальной задержкой

### Обработка ошибок

- **Modbus:** Пойманные исключения → лог → попытка переподключения
- **БД:** Транзакции с rollback при ошибках
- **Веб:** XSS-защита через экранирование, валидация входных данных

### Тестирование

- Явных тестов в проекте не обнаружено (TODO: добавить unit-тесты)
- Для проверки работоспособности:
  - Режим офлайн (`/debug`) — эмуляция без оборудования
  - Логи в `data/logs/` — детальный трассировочный вывод

  - `controller.log` — работа контроллера
  - `owen.log` — Modbus-обмен с ОВЕН

- Пароль БД хранится в `config.yaml` (рекомендуется использовать переменные окружения)
- Клиентские ошибки отправляются на сервер через `/api/client_error`
